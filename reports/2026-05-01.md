## Subagent notes
Subagents were spawned early for reviewer, tester, researcher, and security-review passes.

Observed focus areas from their work:
- **reviewer**: the repo’s strongest trait is still its honesty about what is shipped versus planned, but the next clarity gap moved from plugin framing toward lifecycle-field safety coverage, especially whether risky text hidden outside the main prompt would actually get scanned and surfaced
- **tester**: the local validation surface was green, and the clearest missing regression coverage was around false positives plus hidden-in-metadata prompt-injection text, especially rollback-related fields that looked review-relevant but were not yet explicitly covered
- **researcher**: the best near-term pattern fit was to strengthen observable coverage of existing approval and review metadata before attempting bigger adaptive or enforcement features; broader utility-scoring or memory-loop work still looks promising but not as safe to land in tonight’s budget
- **security reviewer**: the sharpest concrete gap was that prompt-injection scanning covered many text-bearing request fields but not `rollbackPlan`, creating a subtle blind spot in a place operators might trust during review; it also reinforced that the scanner remains heuristic, regex-based, and English-first

## Audit
Main weak spots found today:
- `rollbackPlan` was not part of the prompt-injection text-surface scan, even though rollback notes are still operator-visible instructions that could carry risky hidden text
- the repo had strong bad-case coverage for prompt, summary, and next-action fields, but did not yet prove that rollback metadata was treated with the same suspicion
- the local verification surface still needed a few explicit non-false-positive checks so the security helper stays trustworthy when verbs like "reveal" or "disable" appear in benign contexts
- larger adaptive improvements remain attractive, but the safer high-value move tonight was to harden the current security boundary instead of widening the runtime surface

## Explicit security assessment
### Strengths
- docs, workflows, and source still stay honest that the repo offers classification and review routing, not hard runtime enforcement
- no secrets/auth changes, restart automation, destructive refactors, or external side effects were introduced
- the prompt-injection helper now treats rollback notes as part of the same review surface as prompts, summaries, and next actions
- the new verification path improves trust in both directions: it proves a missed risky surface now escalates, and it proves a couple of benign phrases stay quiet

### Gaps
- prompt-injection classification is still heuristic, regex-based, and English-first
- runtime approval enforcement still does not exist, only classification plus review metadata
- obfuscation, multilingual injection, and broader semantic secret-seeking language are still mostly outside the current scanner’s reach
- proposal and approval metadata remain observable rather than enforceable runtime gates

### Prompt injection posture
Posture improved in a concrete and inspectable way today:
- `src/security.js` now scans `rollbackPlan` alongside the other text-bearing change-request fields
- `src/self-test.js` now asserts that malicious rollback-plan text is caught, while benign phrases like `reveal the design rationale` and `disable the notification sound` stay non-flagged
- the repo now ships an explicit rollback-injection example fixture plus npm script and CI coverage so the new boundary is easy to verify locally
- docs now say more plainly that rollback notes are scanned too, while preserving the warning that the scanner is still heuristic and not hard enforcement

## Research
The strongest research signal today was to keep improving security observability across all review metadata before attempting larger adaptive features.

That translated into a concrete repo-specific choice:
- harden and document an already-important metadata surface, `rollbackPlan`
- add both positive and negative regression coverage so the security helper becomes more trustworthy, not just stricter
- keep bigger work like richer workflow outcome scoring, mutation logs, or deeper memory tuning for later proposal or narrower design passes

This matched the tester and security-review lanes closely, so tonight’s change stayed focused, additive, and reversible.
