## Audit
Main weak spots found today:
- docs still implied more runtime automation than the repo actually ships today
- the boundary between security classification and actual approval enforcement was too easy to misread
- `WORKFLOW-SCORECARD.md` lagged the code by omitting the `security posture` dimension

## Explicit security assessment
### Strengths
- risk classes and approval boundaries remain consistent across docs, workflows, source, examples, and reports
- the prompt-injection helper is visible and locally testable instead of being buried as an implementation detail
- the repo still avoids auth/secrets mutation, restart automation, and destructive refactors
- risky follow-up work was routed into a proposal instead of being forced automatically

### Gaps
- prompt-injection classification is still heuristic, regex-based, and English-first
- `approvalBoundary` is still a routing hint, not a hard runtime gate
- `buildChangeProposal()` can under-classify risky work if a caller skips `evaluateSecurityPosture()` first
- the runtime integration surface is still intentionally thin, so docs need to stay explicit about what is real today

### Prompt injection posture
Current posture remains intentionally modest but clearer:
- direct instruction override, prompt reveal, approval-bypass, environment-dump, destructive, restart, and external-effect paths are exercised locally
- risky prompts still escalate cleanly to `medium-risk` or `approval-required`
- the repo now states more plainly that classification is not the same thing as enforcement
- semantic, multilingual, and indirect file-content injection resistance are still future work, not present-tense guarantees

## Research
The strongest research signal today was about operator trust: approval routing should be visible, auditable, and honest about where enforcement actually happens.

Promising next-step ideas that surfaced:
- a real approval-enforcement and approval-tracking contract
- a combined review artifact or mutation log for better operator visibility
- richer trend tracking for workflow scorecards

Only the first one looked important enough to capture today, and it was routed into a proposal instead of being forced.
