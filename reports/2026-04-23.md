## Subagent notes
Subagents were spawned early for reviewer, tester, researcher, and security-review passes.

Observed focus areas from their work:
- **reviewer**: found the repo strongest where safety language, reports, and terminology stay consistent, but flagged a real gap between the ambitious README/docs story and the still-thin runtime integration surface
- **tester**: validated the runnable surface, found null-input crashes in `scorecard` and `proposal`, and flagged that the CLI error path was still dumping raw file errors instead of a friendlier message
- **researcher**: reinforced that observable safety matters, especially explicit routing and measurable signals; longer-term ideas like drift metrics and mutation logs look promising but are still larger than today’s safe change budget
- **security reviewer**: agreed the security helper is useful as a signal, but called out that it was still heuristic/regex-based and needed both stronger matching and more honest documentation about its limits

## Audit
Main weak spot found: the security posture helper was too easy to under-trust and under-exercise at the same time.

Two concrete problems showed up:
- obvious prompt-injection phrasing like `Ignore previous instructions` was not always matched unless another secret-target heuristic also fired
- the repo had weak resilience around local utility edges (`null` input handling and CLI file-read errors), which made the helper surface less trustworthy during manual inspection

## Explicit security assessment
### Strengths
- clear risk classes and approval boundaries already exist across docs, code, and workflows
- prompt-injection and secret-seeking heuristics already exist in `src/security.js`
- risky categories like secrets, restarts, destructive changes, and external effects still escalate correctly
- the repo continues to avoid auth/secrets mutation and restart automation

### Gaps
- prompt-injection matching was too brittle for a repo that explicitly claims prompt-injection awareness
- the helper output did not expose approval routing as clearly as the proposal path did
- the scanner still needed more explicit documentation that it is a heuristic signal, not a full defense
- the local tool surface had avoidable sharp edges during manual verification

### Prompt injection posture
Posture is better after today’s changes, but still intentionally modest:
- direct instruction-override phrasing now matches reliably
- high-severity injection signals like approval bypass, environment dumps, and secret dumping escalate to `approval-required`
- lower-severity suspicious prompts can still land in `medium-risk`
- the docs now say more plainly that the scanner is heuristic and regex-based, not a standalone security boundary

## Research
A simple pattern still looks right for this repo: if `eclawlution` wants to be trusted as a self-improving companion layer, then security posture should be observable in small local artifacts, and approval routing should be visible in the output, not hidden behind implementation details.

Longer-term ideas from research that look promising but still need review:
- measurable style-drift or persona-drift signals instead of only prose reflection
- append-only mutation logs or similar audit artifacts for safe local changes
- a clearer runtime integration contract so the repo stops implying hooks that do not yet exist
