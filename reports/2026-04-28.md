## Audit
Main weak spots found today:
- `src/security.js` treated any plain mention of token/credential-like terms as `asksForSecrets`, even when the text was a benign documentation request such as clarifying an auth boundary
- that over-classification could incorrectly route harmless local docs/proposal text into `medium-risk`, which is noisy and makes the approval boundary look less trustworthy
- the repo still has the older trust gap around remote publish observability because `origin/main` push access remains blocked for `hunter-st`
- manual wiring is still honest but somewhat under-explained for first-time operators, especially compared with how “plugin-like” the scaffold can look at a glance

## Explicit security assessment
### Strengths
- docs, workflows, and source still stay honest that the repo offers classification and review routing, not hard runtime enforcement
- no secrets/auth changes, restart automation, destructive refactors, or external side effects were introduced
- local verification coverage is broader now because the repo tests a benign `safe-local` security path in addition to the existing medium-risk, approval-required, and non-prompt checks

### Gaps
- prompt-injection classification is still heuristic, regex-based, and English-first
- runtime approval enforcement still does not exist, only classification plus review metadata
- remote push permissions are still blocked, so the repo’s self-publish story remains weaker than the local history suggests
- manual wiring / quick-start guidance can still improve without pretending auto-install exists today

### Prompt injection posture
Posture improved today in a specific and useful way:
- benign auth/token documentation text no longer gets treated as prompt-injection risk just because sensitive nouns appear in the request
- sensitive-material detection now depends on explicit access/exposure language, so approval-required routing stays focused on real reveal/list/dump-style requests
- the repo now ships a `safe-local` security example plus matching self-test/CI coverage, which makes the intended classification boundary easier to verify locally

## Research
The best research signal today matched the concrete security finding: improve trust by reducing classification noise before adding more autonomy.

The strongest near-term pattern fit was to pair sensitive-target detection with explicit access verbs, then prove the boundary with a benign example. That gives operators a clearer local signal without over-claiming enforcement.

The next-best ideas still look promising but proposal-worthy first:
- append-only mutation logs for workflow/proposal state changes
- lightweight approval-state transition tracking with actor + rationale metadata
- more explicit quick-start wiring help for the manual-setup path
