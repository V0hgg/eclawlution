## Subagent notes
Subagents were spawned early for reviewer, tester, researcher, and security-review passes.

Observed focus areas from their work:
- **reviewer**: strongest praise stayed with the repo’s unusually honest shipped-vs-planned separation, then zeroed in on the biggest remaining confusion point, the plugin-shaped scaffold plus thin manual wiring story making the repo look more runtime-complete than it is at first glance
- **tester**: local validation stayed green, breakage risk stayed low, and the main remaining test debt shifted away from today’s docs work toward future negative-case coverage, especially false-positive checks and broader CLI error-path coverage
- **researcher**: the best pattern fit for today was progressive disclosure and reversible onboarding, make the manual wiring path concrete before adding more autonomy, and keep heavier runtime or approval-surface changes in proposal territory
- **security reviewer**: the classifier remains honest but heuristic, regex-based, and English-first; the sharpest safe hardening move today was to reduce operator over-trust by making manual wiring, repo scoping, and plugin limitations much more explicit in user-facing docs and examples

## Audit
Main weak spots found today:
- `openclaw.plugin.json`, `src/index.js`, and the broader phase-2 framing still made the repo easy to misread as an active plugin even though it does not yet register hooks or install jobs
- `docs/USAGE.md` and `examples/openclaw-wiring.md` were still too thin for a first-time operator who wanted to actually wire the daily loops without guessing
- recent reports had been repeatedly flagging the same manual wiring and runtime-surface clarity gap, so shipping a concrete docs/examples pass was higher-value than adding more new promises
- prompt-injection classification logic was already in a decent local state after recent work, so the safer improvement lane today was operator clarity, not more pseudo-enforcement

## Explicit security assessment
### Strengths
- docs, workflows, and source still stay honest that the repo offers classification and review routing, not hard runtime enforcement
- no secrets/auth changes, restart automation, destructive refactors, or external side effects were introduced
- the repo self-evolution path now has a clearer operator-facing boundary between safe local workflow automation and still-manual runtime wiring
- repo self-evolution wiring examples now explicitly route risky, unclear, auth-related, destructive, or surprising changes into `proposals/` instead of implying they should auto-land

### Gaps
- prompt-injection classification is still heuristic, regex-based, and English-first
- runtime approval enforcement still does not exist, only classification plus review metadata
- the docs are now clearer, but the repo still does not ship native hook registration or cron installation
- manual wiring is less confusing now, but it still depends on operator judgment about timezone, delivery, and session binding

### Prompt injection posture
Posture improved at the operator-contract layer today:
- `README.md`, `docs/USAGE.md`, and `docs/PLUGIN-CONTRACT.md` now say much more plainly that the plugin scaffold is descriptive only and does not auto-install or register runtime hooks
- `examples/openclaw-wiring.md` now ships concrete example job objects, including repo-scoped self-evolution language and explicit proposal-routing for risky ideas
- that change does not make the local classifier stronger by itself, but it does reduce a different class of safety risk: users over-trusting automation that the repo does not actually implement yet

## Research
The strongest research signal today was to improve trust through progressive disclosure before expanding autonomy.

That translated cleanly into repo-specific guidance:
- show operators the exact manual wiring shape before implying phase-2 plugin behavior
- prefer reversible examples and scoped prompts over broader automation claims
- keep heavier runtime integration, approval enforcement, and mutation-audit surfaces proposal-first until the repo can support them honestly

This matched the reviewer and security-review lanes closely, so today’s change stayed in docs and examples instead of expanding code scope.
