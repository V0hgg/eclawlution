## Explicit security assessment
### Strengths
- risk classes and approval boundaries remain consistent across source, docs, examples, workflows, and reports
- the repo still keeps approval enforcement honest as advisory/classification rather than pretending it is already hard runtime policy
- no secrets/auth changes, restart automation, or destructive refactors were introduced
- the security helper is locally testable and now more operator-auditable than before

### Gaps
- prompt-injection detection is still heuristic, regex-based, and English-first
- non-prompt fields are still not scanned for injection signals
- approval enforcement is still a caller/orchestrator responsibility, not a built-in runtime gate
- proposal lifecycle state and mutation-log visibility are still future improvements, not solved today

### Prompt injection posture
Posture improved in a modest but concrete way today:
- matched prompt-injection signals now carry severity metadata
- direct secret-reveal requests now escalate as high-severity prompt-injection instead of relying only on broader secret-target heuristics
- security evaluation output now exposes structured `blockerDetails` plus a `handlingRecommendation`, making review routing easier to audit locally
- the repo still stays honest that this is classification support, not standalone defense or hard enforcement

## Research
The strongest research signal today was about operator trust through observable audit trails, especially mutation logs and proposal lifecycle state.

Those still look promising, but they are a broader surface than today’s safest local improvement. For this cycle, the better tradeoff was to improve the security helper’s observability and tighten its tests rather than adding new workflow artifacts without a clearer contract.

## What changed
- hardened `src/security.js` so prompt-injection signals now include severity metadata instead of only ids/descriptions
- added high-severity detection for direct secret-reveal phrasing like exposing API keys or credentials
- made `evaluateSecurityPosture()` resilient to `null` / non-object input
- added structured `blockerDetails` to security evaluation output so blockers carry severity, source, summary, and evidence
- added `handlingRecommendation` to security evaluation output so review routing is easier to interpret without implying hard enforcement
- expanded `src/self-test.js` to cover:
  - package/manifest version consistency
  - exact scorecard overall calculation
  - high-severity signal metadata
  - direct secret-reveal escalation
  - null security-posture handling
  - new blocker-detail / handling-recommendation fields
- updated `README.md`, `docs/USAGE.md`, `docs/SECURITY-MODEL.md`, and `docs/PLUGIN-CONTRACT.md` so the documented security surface matches the new observable output
